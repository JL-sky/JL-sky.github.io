容器命令

docker run

使用方式 作用 选项说明
docker run -it 镜像名称 /bin/bash 以前台交互模式进入容器 -i，即interaction，以交互模式运行容器 -t，即terminal，给容器分配一个伪终端 /bin/bash：指定给容器分配的伪终端为bash
docker run -it 镜像名称 –name 容器名 /bin/bash 以前台交互模式进入容器，并给容器命名 –name：给容器重新命名
docker run -d 镜像名称 以后台方式进入容器给

退出容器

退出方式1

使用exit命令或者快捷键ctrl+d

使用这种方式退出容器后，容器本身也会停止

退出方式2（建议使用）

使用快捷键ctrl+p+q

使用这种方式退出容器后，容器仍旧在运行，不会停止

重新启动正在运行的容器

使用方式 作用
docker start 容器id 重新启动正在运行的容器

停止正在运行的容器

使用方式 作用
docker stop 容器id 停止正在运行的容器

删除容器

使用方式 作用
docker rm 容器id 删除正在运行的容器
docker rm -f 容器id 强制删除容器，由于使用docker rm命令删除容器不能删除正在运行的容器，需要首先停止容器的运行才能删除，如果需要删除一个正在运行的容器可加上-f选项强制删除，但不推荐
docker rm -f $(docker ps -qa) 强制删除所有容器
docker ps -qa | xargs docker rm 删除所有容器

重新进入正在运行的容器

docker exec（建议使用）

使用方式 作用
docker exec -it 容器id /bin/bash 重新进入正在运行的容器，并以交互模式进入

注， 使用这种方式重新进入容器后，即使使用exit命令退出容器，容器本身也不会停止

docker attach

使用方式 作用
docker attach 容器id 重新进入正在运行的容器

注， 使用这种方式重新进入容器后，即使使用exit命令退出容器，容器本身同时也会停止运行

容器的导出与导入

容器导出

容器导出就是指将容器进行打包归档为tar文件以备份，防止有人误删容器

命令使用格式为

1

docker export 容器ID > 归档名.tar

如下，我们对pcl/ubuntu容器进行导出，其中55d6651e8842是pcl/ubuntu的容器id

1
2
3

pcl@robot:~/Public$ docker export 55d6651e8842 > ubuntu1.2.tar
pcl@robot:~/Public$ ls
ubuntu1.2.tar

容器导入 

1

docker import ubuntu1.2.tar pcl/ubuntu:1.2

容器导入就是指将打包归档为tar文件的备份容器进行“解包”，并重新导入为镜像文件

该命令的使用格式为

1

docker import 归档文件 镜像名:TAG

镜像的保存与加载

镜像保存

作用是将镜像文件保存为归档文件，注意与容器导出进行区分，容器导出是将容器保存为归档文件

命令使用方式

1

docker save 镜像id -o 归档名.tar

我们使用刚才导出的镜像文件进行实验

1

docker save pcl/ubuntu:1.2 -o ubuntu.1.3.tar

镜像加载

作用是将保存的镜像归档文件重新解包为docker镜像文件

使用方式为

1

docker load i 镜像打包文件

如下所示，使用docker load命令看是否能加载出镜像文件

1

 docker load -i ubuntu.1.3.tar

对比总结我们发现

  • 容器导出是将容器保存为归档文件
  • 镜像保存是将镜像保存为归档文件
  • 无论是将容器保存为归档文件还是将镜像保存为归档文件，使用对应命令解包（如容器导出对应的是容器导入，镜像保存对应的是镜像加载）之后，都只是解包为镜像文件
  • 日常情况下，我们常使用的是docker save 与 docker load 也就是镜像的保存与加载

容器文件拷贝到本地

使用方式 作用
docker cp 容器id:容器内文件路径 本地路径 拷贝容器内的文件到本地

同理，我们更换源路径与目标路径也可以实现将本地文件拷贝到容器中

从容器创建新镜像

我们知道，从docker-hub上拉取的镜像所创建的容器是最小版本的，比如ubuntu内部是没有vim编辑器的，我们需要自己手动安装，但是当我们安装后假如有人把我们的容器误删了，如果我们重新根据原始镜像创建容器，那么这个由原始镜像创造的容器就没有了我们之前配的环境，这意味着我们需要重新配置环境。。。。。

那么，有什么办法解决这种困境呢？

一般情况下，我们有两种方法制作自己的镜像

  1. 重写dockerfile文件
  2. 在原来容器的基础上配好环境之后，使用docker commit命令制作

有关dockerfile的说明我们在下节说明，今天我们介绍如何使用docker commit命令将一个新的容器制作成一个镜像

制作新容器

我们以最小版的ubuntu镜像为例，并创建一个ubuntu容器，在这个容器里安装vim（原始的ubuntu容器是没有vim的），之后使用docker commit命令根据这个容器创建一个新的镜像，如果我们制作成功，那么我们使用新的镜像创建容器后，容器会有vim

1

 docker images

如下，我的docker镜像里有一个原始的ubunutu镜像，该镜像的大小只有72.8MB

1
2
3

4m                       latest           23a2509258e9   5 weeks ago     18.1GB
ubuntu                   20.04            6013ae1a63c2   7 weeks ago     72.8MB

接下来我们根据这个镜像创建一个容器并进入

#如下已经进入容器了
root@ce501a8563f7:/#

  • –name选项表示我们给创建的容器取一个名字，否则每次我们创建的容器名字就都是随机的

制作新镜像

接下来，我们根据这个容器重新制作一个新的镜像，使得下次我们使用这个新镜像创建容器

我们先给出docker commit命令的使用格式，但我们一般用不到这么多参数

1

docker commit -m "提交信息" -a="作者名" 容器ID 要创建的目标镜像名:版本号

1.退出容器

1
2

root@ce501a8563f7:/# exit
(base) jl@jin-X299X-AORUS-MASTER:~$

2.根据该容器制作新镜像

1
2

(base) jl@jin-X299X-AORUS-MASTER:~$ docker commit ubuntu_vim ubuntu_vim_img
sha256:cedfbe36057f927d1b6deca0667a30ce5214076121eca0547055260d3d91be0b

其中

  • ubuntu_vim，是我们刚才新容器的名字，如果你没有给容器命名，可以使用容器ID代替
  • ubuntu_vim_img，是我们用这个容器制作的镜像名称，自己取就好

如下所示，我们重新查看镜像列表，已经有了我们创建好的新镜像

1
